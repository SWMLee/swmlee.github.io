angular http请求路径IP地址重复
发表于 2020-08-16 20:09:30| 更新于 2020-09-07 22:05:39|技术文章angular
字数总计:1.7k|阅读时长: 5 分钟

现象描述

写了一个 angular 项目，测试多次都没问题，就要打包部署到云端服务器去，部署完成之后，打开页面访问，部分数据资源都查不到结果，之后返回预设的访问未经授权地址的报错信息。

我都用 admin 账号了，自然所有后台接口都应该是畅快访问的，为什么会权限问题呢？一个简单的 get 请求，没多少严苛要求啊，不用想，是不是路径错了，打开控制台，果然，请求地址里面，有两个 IP 地址，类似下图：

虽然一开始我想直接开始甩锅，毕竟本地测试那么久都没问题，现在部署上去，怎么会突然出现问题呢？但是这个是请求地址有误啊，莫名奇妙两个 IP，怎么看都是前端的问题。我就想说用域名啊，可惜说域名不稳定，经常 ping 不同，后续再解决。

我们几个人就看了很久，能想到请求相关的就请求数据的 service、http 拦截器等等地方，看起来都很正常。IP 地址出现的文件就一个，就两行，地址看起来也正确，过网关什么的 path 也有加。到底哪里出的问题？

因为咋看之下，只是请求的 IP 地址重复了，还不一定把锅甩给谁，虽然知道是大概率是自己的，但是还是去排查看看是不是部署的容器问题了、静态资源服务器配置的问题了等等。

其实这类问题有个查锅最快的方式，就是抓包，哪里有问题一眼就出来，不出所料，请求在出去的时候，就已经 IP 重复了，根本不是后台的锅。

怎么办？静态资源部署到云端，后台也上云了，我本地运行没办法连后台（原因不想说），居然就在控制台又美化压缩后的 js，找个有拦截到请求随意地点，跳到请求处理相关的地方。从结果上来看，从 http 请求创建，到透过拦截器发出，看起来都是正常的，但一些更多的事件回调之类的地方就看不到了。

一度有点兴奋。

愚蠢的细节错误

那么问题出在了哪里呢？

再看看这片段几个人看了无数次的地方，就类似：

虽然单独截图出来了，还有放大，可以一眼就看到了。但是编辑器小小又多的内容，几个人看了好多次，都没曾想到会是这里的问题。

http 请求后面的左斜线少了一个。

就是这么一个小小的问题。这是我才想起来，上云前有改动的唯一地方，就是修改了 IP 地址，可能就是那个时候 ctrl c ctrl v 的时候，不小心多划了一个。

可笑是可笑，问题是解决了，但是为什么呢？常识性分析就是因为只有一个/，angular 应用就把这个/IP 当成了一个普通 path，解析 url 发现有协议，有端口，有路径，但是没有 host，就自动补入了应用所在的 host。

这就是 angular 的“周全”之处，它可能认为你是故意这样写的，就会按照一定的规则，帮你重塑正是的请求。看起来它把输入的 url 字符串按照一定规则进行截取，第一个冒号前的，算是协议，有/分割的当成了 path 路径，处理完之后没有找到 ip，就抓取当前应用所在的网域当成请求 IP。

那么正常的，就是// 之后的就是 IP 了。事实上，url 中，“http://”这整个部分，才是 protocol，而不仅仅只是“http”几个字符就是协议了。
